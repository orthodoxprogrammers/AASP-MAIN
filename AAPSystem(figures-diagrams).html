1
Sprint 1 — Foundation & Authentication
Weeks 1–4 · Project setup, database design, login system, role-based access
▾
Sprint Goal

Establish the technical foundation — database schema, user authentication, and role-based routing so both MSWDO and Barangay users can securely access their respective dashboards.

User Stories
  • As an MSWDO admin, I can log in and be redirected to the main dashboard
  • As a Barangay user, I can log in and be routed to my barangay's household view
  • As a developer, I need a normalized DB schema covering all 16 tables
  • As an MSWDO admin, I can see total beneficiary stats on the dashboard
  • As any user, I am blocked from accessing pages outside my role (Access Denied)
Tasks Completed
  • Database design: 16-table schema in mswdo_aps_db (MariaDB 10.4)
  • Seed data: 12 user accounts (1 MSWDO + 11 barangay accounts)
  • login.php + auth.php with password_verify() and session management
  • RBAC middleware: role check on every protected page using $_SESSION['role']
  • dashboard.php: COUNT / SUM / DISTINCT aggregate queries for KPI cards
  • Base layout: sidebar.php + topbar.php for both MSWDO and Barangay views
  • config/db.php: mysqli connection with error handling
Scrum Ceremonies
Sprint Planning — Define DB schema with MSWDO stakeholder
Daily Standups — 3x per week syncs
Sprint Review — Demo login + dashboard to MSWDO
Retrospective — Clarify field requirements for intake form
Deliverables
login.php auth.php dashboard.php mswdo_aps_db schema Role-based layout (sidebar + topbar) 12 seeded user accounts
2
Sprint 2 — Beneficiary Management Core
Weeks 5–8 · Add, view, edit, search, duplicate detection, sectors & family
▾
Sprint Goal

Build the full beneficiary lifecycle — registration with multi-step form, duplicate detection, sector tagging, family composition, and priority scoring — for both MSWDO and Barangay roles.

User Stories
  • As MSWDO, I can register a new beneficiary with client info, sectors, and family members in a multi-step form
  • As Barangay, I can add a beneficiary restricted to my assigned barangay only
  • As any user, I am warned before saving if a similar name already exists
  • As MSWDO, I can search, filter, and paginate the beneficiary list
  • As any user, I can view a detailed beneficiary profile with all linked data
  • As MSWDO, I can edit any beneficiary; Barangay can only edit their own
Tasks Completed
  • add_beneficiary.php: multi-step JS stepper (step 1: client, step 2: beneficiary, step 3: GIS)
  • check_beneficiary.php: AJAX duplicate detection with LIKE-query, returns JSON
  • save_beneficiary.php: full transaction — clients → beneficiaries → general_intake → sectors → subcategories → family_composition
  • beneficiary.php + beneficiary_table.php: paginated list with search & barangay filter
  • barangay/add_beneficiary_barangay.php: same flow with WHERE barangay isolation
  • view_beneficiary_info.php: full profile view with intake history, sectors, family
  • edit_beneficiary.php + update_beneficiary.php: update all linked tables in one transaction
  • priority_score computation logic integrated into save/update flow
Scrum Ceremonies
Sprint Planning — Backlog refinement for GIS fields with MSWDO staff
Daily Standups — 3x per week
Sprint Review — Demo full add/edit/view flow to MSWDO and one Barangay user
Retrospective — Feedback: add real-time age auto-compute from birthdate
Deliverables
add_beneficiary.php save_beneficiary.php check_beneficiary.php beneficiary.php beneficiary_table.php view_beneficiary_info.php edit_beneficiary.php Barangay equivalents (add/edit/view/list)
3
Sprint 3 — Intake Workflow, Archive & Print
Weeks 9–12 · General Intake approval flow, archive/restore, DSWD A4 print forms
▾
Sprint Goal

Deliver the approval workflow (Pending → Approved → Released), archive/restore capability, and the full 4-page DSWD print system so MSWDO staff can generate official documents directly from the system.

User Stories
  • As MSWDO, I can approve or release a beneficiary's intake request and record the amount released
  • As MSWDO, I can archive a beneficiary with a reason; the record is preserved and restorable
  • As MSWDO, I can view all archived beneficiaries in a separate list and restore any
  • As MSWDO or Barangay, I can print all 4 DSWD forms pre-filled with the beneficiary's live DB data
  • As a Barangay user, I can print the household list for my barangay sorted by priority
Tasks Completed
  • general_intake.status workflow: Pending → Approved → Released → Cancelled
  • archive_beneficiary.php + process_archive_beneficiary.php: status toggle with archived_reason / archived_at / archived_by
  • archived_beneficiaries.php: filtered list of archived records for MSWDO
  • restore_beneficiary.php: sets status back to active
  • print_beneficiary_barangay.php: 4-page A4 PHP print renderer (Info Sheet, GIS p1, GIS p2, Certificate)
  • Logos embedded as base64 via imgBase64() function (no server path dependency)
  • All GIS sections (I–XI) wired to actual DB columns; checkboxes populated from comma-separated text fields
  • print_households_barangay.php: landscape A4 household list with priority rank and zebra striping
  • print_beneficiaries.php: MSWDO-level sorted print list across all barangays
Scrum Ceremonies
Sprint Planning — Map DSWD paper form fields to DB columns; confirm print layout with MSWDO
Daily Standups — 3x per week
Sprint Review — Print test on A4 paper; MSWDO verified all 4 pages match official DSWD format
Retrospective — Bug fix: page 4 mode checkboxes used wrong column; fix applied sprint-end
Deliverables
archive_beneficiary.php process_archive_beneficiary.php archived_beneficiaries.php restore_beneficiary.php print_beneficiary_barangay.php (4-page A4) print_households_barangay.php print_beneficiaries.php Intake status approval flow
4
Sprint 4 — Testing, Refinement & Deployment
Weeks 13–16 · UAT, bug fixing, performance tuning, final deployment
▾
Sprint Goal

Conduct User Acceptance Testing (UAT) with actual MSWDO staff and barangay encoders, resolve all discovered bugs, optimize queries, and deploy the system to the local government server running XAMPP (Apache + MariaDB).

User Stories
  • As MSWDO staff, all workflows should perform correctly on real beneficiary data without errors
  • As a Barangay encoder, I cannot view or modify records from other barangays
  • As MSWDO, the priority score correctly ranks beneficiaries with lower income higher
  • As any user, the print forms display correctly when printed from the browser
  • As a developer, all DB queries use prepared statements to prevent SQL injection
  • As stakeholders, system is accessible on the local government network via XAMPP
Tasks Completed
  • UAT sessions: MSWDO staff tested add/edit/print/archive with real beneficiary data
  • Bug fix: print_beneficiary_barangay.php — page 4 onsite/offsite used $visitType string (wrong); fixed to use $isOnsite/$isMalasakit/$isOffsite booleans
  • Bug fix: sections III–VII checkboxes were hardcoded disabled; wired to actual comma-separated DB fields
  • Added DB indexes: idx_barangay, idx_sex, idx_age, idx_identity for pagination performance
  • Input sanitization audit: all $_POST values run through formatName() or trim()
  • Cross-browser print test: Chrome + Edge verified for A4 @page CSS output
  • Final deployment: mswdo_aps_db imported to production XAMPP; all 12 barangay accounts verified
  • System documentation completed: README, APS structure.txt, ERD, Use Case Diagram

Sprint Planning — Prioritize UAT bug backlog with MSWDO
Daily Standups — 3x per week
Sprint Review — Final live demo to MSWDO head and barangay captains
Retrospective — Post-mortem; identify future enhancements (e.g., SMS notifications, online access)
Deliverables
